## Purpose

These documents help:
- **Track development progress** across multiple sessions
- **Document compliance** with safety-critical guidelines
- **Record architectural decisions** and their rationale
- **Provide context** when resuming work after breaks
- **Maintain institutional knowledge** of the codebase evolution

## Document Lifecycle

1. **Active Development** - Documents updated frequently during work sessions
2. **Completed Work** - Archived here when features are finished
3. **Historical Reference** - Used to understand past decisions and evolution

## Adding New WIP Documents

When creating new tracking documents:
- Use clear, descriptive names indicating content type
- Include date stamps for time-sensitive tracking
- Update this README with a brief description
